QME Runner Extensions
=====================

This note explains how to add new optimization/search runners (local or two-ended)
without changing core logic.

Background
----------
The Explorer uses a pluggable strategy registry. Strategies are small callables
that accept either a single Atoms (local) or a list/sequence of Atoms (two-ended),
and return whatever result is appropriate (typically an Atoms or list of Atoms).
Explorer injects `explorer=self` and standard defaults into the runner call.

Where to put code
-----------------
- Local runners:   qme/core/local_strategies.py
- Two-ended runners: qme/core/twoended_strategies.py

You can add your runner in either file or in a new module and import it in Explorer
before registration.

Runner signature
----------------
- Local runner:
    def my_local_runner(atoms, fmax=0.05, steps=1000, explorer=None, local_optimizer_name="sella", **kwargs):
        # attach calculator via explorer helper
        explorer._create_and_attach_calculator(atoms)
        explorer._apply_constraints(atoms)
        # run optimization ...
        return atoms

- Two-ended runner:
    def my_twoended_runner(atoms_list, npoints=11, explorer=None, **kwargs):
        # atoms_list: [reactant, product] or longer sequences
        # build path, refine, return result(s)
        return result

Registering the runner
----------------------
In Explorer.__init__, register your runner with a namespaced key and an alias:

    from qme.core.local_strategies import my_local_runner
    self.register_strategy(
        "local:afir", my_local_runner, strategy_type="local",
        description="AFIR-based local optimization", aliases=["afir"]
    )

    from qme.core.twoended_strategies import my_neb_runner
    self.register_strategy(
        "twoended:neb", my_neb_runner, strategy_type="two-ended",
        description="NEB path refinement", aliases=["neb"]
    )

Selection & CLI
---------------
- Explorer.run() prefers namespaced keys but keeps backward-compatible aliases.
- The CLI can expose new sub-mode flags or allow `--runner-name` to select custom
  registered runners directly, e.g. `--runner-name twoended:neb`.

Testing
-------
- Add unit tests under tests/ exercising the new runner with the mock backend.
- Keep steps small (1–2) to speed up CI.

Tips
----
- Prefer clear, small runners over complex multi-purpose ones.
- Reuse Explorer helpers for calculator/constraints to keep behavior consistent.
- Document your runner via the `description` parameter during registration.
